# ADP schemas

Canonical JSON Schema (**draft 2020-12**) for the Action-data package format.
Each file below defines one piece of a package; the manifest ties them together.

| Schema | Layer | Defines |
|--------|-------|---------|
| [`manifest.schema.json`](manifest.schema.json) | — | Package root `adp.manifest.json`: `adp_version`, `session_ref` (uuid), `producer`, `scrub`, `parts[]` (each with BYOS `location`, `format`, `fallback`, `bytes`, `hash`), and `tiers{ light, heavy }`. |
| [`session.schema.json`](session.schema.json) | 2 | Session descriptor: span, `activity_hint`, `producer`, and `source_summaries[]` — per-source device aggregates not re-derivable from the timeline. No owner/visibility/status. |
| [`timeline.schema.json`](timeline.schema.json) | 2 | ONE row of the 1 Hz UTC timeline. `required: [time_utc]` only; nullable `lat`/`lon`; canonical columns; `grit`/`flow`; `patternProperties` for `<source>_<metric>` shadow columns, `<source>_<lat\|lon>` shadow GPS, and `<source>_raw` presence flags. |
| [`provenance.schema.json`](provenance.schema.json) | 2 | `sources[]` (per raw source: device, file, window, clock-sync, dual-HR anchor, **sha256** content hash) + `canonical` (per-column source, gap classification `source_silent`/`field_missing`, coverage %, longest gap) + optional `diagnostics` (cross-source merge-quality facts: `source_overlap_s`, `hr_mean_abs_diff_bpm`, `video_sync`, `warnings`). |
| [`clips.schema.json`](clips.schema.json) | 3 | The **clips part**: an array of first-class clip objects. Each places a footage segment on the canonical timeline (`start_utc`/`end_utc`/`duration_s`), references the video asset (`video_ref`), carries the video↔telemetry sync offset (`offset_s` **on the clip**), the generalized video technical block (`video.{file,width,height,fps,codec,size_bytes,recording_start_utc}`), and optional `direction`/`intent`/`labels`/`description`/`derived_metrics`. Sport-neutral. |
| [`enrichment.schema.json`](enrichment.schema.json) | 3 | One attributed enrichment: `anchor`, `kind`, `attribution{author, author_kind, produced_utc, trust, ...}`, `payload`. The **clip** payload contract (`$defs/clip`) models a user-authored highlight/annotation clip (possibly multi-range); the footage-derived clips part is `clips.schema.json`. |
| [`storage-map.schema.json`](storage-map.schema.json) | — | **Not part of a package.** A user-held map from logical store aliases (`default`, `cold`, `byos:<alias>`) to real endpoints. Secrets are never inline — a store references credentials by name (`credentials_ref`). Used by the reader to resolve `parts[].location.store`. |

## The timeline is parquet in practice

The canonical timeline is produced and stored **column-wise as parquet**
(`derived/timeline.parquet` in producer output), not as JSON. It is far too large
and too regular for per-row JSON. `timeline.schema.json` therefore documents the
**row/column contract** — field names, types, nullability, and the
`<source>_*` column families — that the parquet columns must satisfy. Use it to
validate a decoded row (or a JSON export) and as the authoritative column
reference; do not expect timeline parts to ship as JSON.

## Identity, encoded in every part

- **`session_ref`** — a stable **UUID** for the logical activity, repeated in the
  manifest, session, provenance, and enrichment schemas. Survives the user
  appending new raw data later.
- **Per-source sha256** — `provenance.sources[].hash` (pattern
  `^sha256:[0-9a-fA-F]{64}$`, required) for dedup / integrity / duplicate-import
  detection. Distinct from each part's own `hash` in the manifest.

## Design notes carried by the schemas

- **Nullable `lat`/`lon`** — GPS can be absent for a whole second (dropout or
  pre/post source span); every telemetry column is nullable, only `time_utc`
  is required.
- **Shadow columns** — `<source>_<metric>` (e.g. `bosch_hr_bpm`, `garmin_alt_m`)
  preserve each source's own value even when canonical picked a different source;
  `<source>_raw` marks real vs interpolated/filled samples.
- **Facts vs projections** — the schemas encode only facts. Derived metrics,
  map-match, overlays, and HLS are projections and have no schema here: they are
  regenerated by consumers, not carried in the package.
- **Soft segment refs** — geo/segment references are soft text refs
  (`<wayId>-<index>`), not foreign keys; see the enrichment `segment` kind.
