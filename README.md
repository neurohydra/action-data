# Action-data package (ADP)

**ADP** is a portable, self-describing container format for action-sports
activities: the merged telemetry, the video references, and every layer of
enrichment built on top — packaged so it can be moved, shared, backed up, and
re-used independent of any single service.

This repository is the **canonical home for the ADP format**: the JSON Schema
that defines the format, and the reference Python tooling (`adp`) for wrapping
producer output into a package and validating it. It is deliberately
project-independent — the producer pipeline
([`randomfinnrides-pipeline`](../randomfinnrides-pipeline)) and consumer apps
(e.g. live-trails) depend on this format, not the other way round.

## The 3-layer model

An activity is captured and refined in three layers with different lifetimes and
egress rules:

| Layer | Name | Egresses? | What it is |
|-------|------|-----------|------------|
| **1** | raw | **No** (immutable, stays at origin) | Original device/camera files: FIT (Garmin, Bosch), Bosch logs, video (`.mp4`). Never scrubbed, never leaves. |
| **2** | refined canonical | **Yes** (scrubbed) | One UTC timeline; all sources merged onto a 1 Hz grid with per-column provenance. The portable core of the package. |
| **3** | enrichment | **Yes** (additive, attributed) | Clips, external metrics, OSM difficulty match, weather. First- or third-party, each attributed. |

### Facts vs projections — the boundary that decides what travels

The package carries **facts**: things that are *not re-derivable* from other
package contents. It does **not** carry **projections**: anything a consumer can
recompute on demand.

- **Facts (travel in package):** the merged 1 Hz timeline, per-source device
  aggregates that the device reported but the 1 Hz timeline can't reproduce
  (calories, training load, grit/flow, laps, respiration, Bosch avg/max
  power+cadence), raw-source provenance + content hashes, clips, third-party
  enrichment.
- **Projections (cache, do NOT travel):** derived summary metrics, map-match
  results, rendered overlays, HLS/proxy transcodes. Regenerated by consumers.

**Service metadata is not in the package.** Ownership, visibility, status, and
other service-account concerns belong to whatever service hosts a package, not
to the package itself. A package is portable precisely because it does not
encode who owns it.

## Anatomy of a package

A package is a self-describing manifest (`adp.manifest.json`) plus a set of
logical **parts**. Each part is resolvable from any storage back-end (BYOS,
below) via its `location` + `format` + `hash` + `bytes`.

```
adp.manifest.json          # root: version, session_ref, parts[], tiers
  parts[]
    session      (layer 2)   session_ref, span, activity_hint, source_summaries[]
    timeline     (layer 2)   1 Hz UTC rows (stored as parquet in practice)
    geo          (layer 2)   redacted/generalized share track (GeoJSON/GPX)
    provenance   (layer 2)   per-source raw origin + sha256, per-column coverage
    clips        (layer 3)   first-class clip objects
    <enrichment> (layer 3)   weather / osm_difficulty / external_metric / ...
    raw          (layer 1)   original files — heavy tier only
    video        (layer 1)   original footage — heavy tier only
```

### BYOS — bring your own storage

A part's bytes live wherever the owner puts them. `location.store` is one of:

- `default` — the service's own default store.
- `byos:<alias>` — a user-supplied store (their own S3/R2/bucket), referenced by
  a named alias resolved out-of-band.
- `cold` — archival / cold storage; may need rehydration before access.

Because every part carries its own `hash` and `bytes`, a package stays verifiable
and reassemblable no matter which store each part came from.

#### Resolving aliases — the storage map (never in the package)

The manifest carries only the **logical alias**, never an account, bucket URL, or
credential — so the package leaks nothing about where its owner keeps their bytes.
A reader turns an alias into a real endpoint with a **storage map**: a separate,
user-held file (schema: [`schemas/storage-map.schema.json`](schemas/storage-map.schema.json))
that maps each alias → `{ kind, endpoint, credentials_ref?, … }`. Secrets are
**never inline** — a store names its credential (`credentials_ref`, e.g. an env
var or cloud profile), resolved out-of-band.

The reference resolver is [`adp/store.py`](adp/store.py):

- `default` with **no** map → the historical local behavior: the part key is a
  local path (absolute as-is, relative under the package dir). Default packages
  work with no map, unchanged.
- any alias present in the map → resolved via its entry. A `kind: local` entry
  yields a local filesystem path; remote kinds (`s3`/`r2`/`gcs`/…) yield a URI
  only (not fetchable by this reference tooling).
- `byos:<alias>` / `cold` with no map entry → unresolvable (error).

`adp validate` and `adp project` take an optional `--storage-map <file>`; without
it, default local packages behave exactly as before.

```
adp validate out/<ride> --storage-map my-stores.json
adp project  out/<ride> --to ridepackage-v0 --storage-map my-stores.json
```

### Packaging tiers

- **`light`** = `{ session, timeline, geo, provenance, clips }` — the shareable,
  scrubbed, storage-cheap core. This is what egresses by default.
- **`heavy`** = `{ raw, video }` — the layer-1 originals. Large, private, usually
  stays at origin; included only for full backup/transfer.

## Identity

- **`session_ref`** — a stable **UUID** identifying the *logical activity*. It
  survives the user appending new raw data later (a second FIT file, a second
  camera): the session is the same activity, so the ref does not change.
- **Per-source content hash** — every raw source in `provenance.sources[]` carries
  a **sha256** of its original bytes, for dedup, integrity checks, and detecting
  duplicate imports of the same source file.

## Reference tooling

The `adp` package ships a CLI (entry point `adp`) with three subcommands:

- `adp wrap` — wrap producer output (e.g. an rfr ride folder) into an ADP-light
  package. Maps the producer's per-segment video block onto canonical clip
  objects (`clips.schema.json`) and carries cross-source diagnostics into
  `provenance.diagnostics`.
- `adp validate` — validate a package against the schemas (manifest, session,
  provenance, **clips**, timeline) and re-verify content hashes. Takes an
  optional `--storage-map`.
- `adp project --to ridepackage-v0` — project a package back into the live-trails
  RidePackage v0 consumer folder; the canonical clips map back **byte-faithfully**
  to the v0 `clips[]`. Takes an optional `--storage-map`.

## Schemas

The JSON Schema files (draft 2020-12) live in [`schemas/`](schemas/); see
[`schemas/README.md`](schemas/README.md) for the index. The timeline is stored
as **parquet** in practice — its schema documents the per-row/per-column
contract.

## Grounding

The schema is grounded in real producer output from `randomfinnrides-pipeline`:
an rfr ride folder (`manifest.json` with a `ride` block, `derived/timeline.parquet`,
`video/`, `validation.json`). ADP's `session` part generalizes rfr's `ride`
block; rfr's `timeline.parquet` columns are the timeline contract; rfr's
`validation.json` coverage output is the `provenance.canonical` contract.
